# Plan 02 — Products In/Out (create flows + line editors)

## 🧠 Architecture explanation

This plan migrates the **inventory movement** parts of the Product feature into `shadcn-isp-client`.

Key decisions (aligned with `migrate-feature.md`):

- **Server by default, client only where needed**: pages remain server; the forms/line editors are client.
- **No Flowbite**: replace all Flowbite UI with existing shadcn-based components.
- **Form pattern**: use `FormBuilder` + `FormWrapper` for header fields and submission, but implement the `product[]` line array with a **custom “invoice-lines-editor” style component** (same pattern as `[components/invoices/invoice-lines-editor.tsx](C:/wamp64/www/shadcn-isp-client/components/invoices/invoice-lines-editor.tsx)`). This is required because the lines contain dependent dropdowns, derived totals, serial/fiber conditional UI, and per-line effects.
- **Data fetching + mutation**: use `useFetch` via `useApiQuery` / `useApiMutation` (already the standard in this repo). No Next API proxy routes will be added.
- **Strict typing**: all form state and payloads are validated with Zod; payload transforms normalize dates/numbers similar to `[components/invoices/invoice-type.ts](C:/wamp64/www/shadcn-isp-client/components/invoices/invoice-type.ts)`.

Plan-01 dependency:

- Assumes `/products` (CRUD listing) exists (from plan-01) and you can navigate back to it after successful In/Out creation.

### Product In types (`product-in-type.ts`)

Define:

- **Ref schemas** (for any nested display objects if needed later):
  - `StaffRefSchema` (id, name nullable)
  - `FundRefSchema` (id, name nullable)
  - `VendorRefSchema` (id, name nullable)
  - `ExpenseTypeRefSchema` (id, name nullable)
- **Line schema** `ProductInLineSchema`:
  - `product_id` (required)
  - `product_category_id` (optional)
  - `brand` (nullable)
  - `status` enum: `new | old | replace` (default `new`)
  - `quantity`, `unit_price`, `unit_sell_price`, `vat`, `total_price` (numbers; coerce + normalize)
  - `has_serial` (0/1)
  - `serial`: array of strings (serial numbers)
  - fiber fields: `fiberID`, `fiber_meter_start`, `fiber_meter_end` (optional/nullable)
- **Expense schema** `ProductInExpenseSchema` (nested under `expense`):
  - `expense_type_id` required
  - `vendor_id`, `fund_id` optional
  - `amount` derived from total (still send; keep in payload)
  - `note` optional
- **Form schema** `ProductInFormSchema`:
  - `purchase_by` required (staff id)
  - `purchase_date` optional date
  - `voucher` optional
  - `note` optional
  - `expense` optional (but if present, must contain required `expense_type_id`)
  - `product`: `ProductInLineSchema[]` with min(1)

Validation refinements:

- **Serial count/empties**: if `has_serial === 1`, require `serial.length === quantity` and every serial string is non-empty after trim.
- **Fiber quantity**: if fiber is used (`product_category_id === 2`), quantity is derived from meters (UI enforces, schema can also ensure `quantity >= 0`).

Payload transform (required):

- Convert any Date objects to API date strings (`YYYY-MM-DD`) like invoice uses.
- Normalize all numeric fields using a `toNumber`-style coercion.
- Ensure `total_price` matches UI computation (
  - base: `unit_price * quantity`
  - if `vat > 0`: add VAT percentage
  ).

Export:

- `ProductInFormInput`, `ProductInPayload`, `ProductInLine`.

### Product Out types (`product-out-type.ts`)

Define:

- **Line schema** `ProductOutLineSchema`:
  - `product_id` required
  - `productin_id` optional/required depending on backend rule (treat as required when product is serial-managed or when backend needs stock source)
  - `quantity`, `unit_price`, `vat`, `discount`, `total_price`
  - `has_serial` (0/1)
  - `serial`: array of selected serial ids (numbers)
  - fiber fields (optional)
- **Form schema** `ProductOutFormSchema`:
  - `out_date` optional date
  - `received_by` required (staff id)
  - `client_id` optional
  - `create_invoice` (0/1 default 0)
  - `note` optional
  - `product`: `ProductOutLineSchema[]` min(1)

Validation refinements:

- If `has_serial === 1`, require at least 1 serial selected; if backend expects exact quantity match, enforce `serial.length === quantity`.

Payload transform:

- Normalize numbers/dates.
- Compute totals:
  - base: `unit_price * quantity`
  - add VAT percentage
  - subtract `discount`.

Export:

- `ProductOutFormInput`, `ProductOutPayload`, `ProductOutLine`.

## 🔄 Refactored API logic (endpoints to use)

Old project used Next API proxies like `/api/product-ins` → `${BASE_URL}/api/v1/products-in`.
New project must call backend directly through `useFetch`/`useApiMutation`:

- Create Product In: `POST /products-in`
- Create Product Out: `POST /products-out`
- Product detail (for line autofill): `GET /products/{id}`
- Dropdowns:
  - `/dropdown-products`
  - `/dropdown-staffs`
  - `/dropdown-clients`
  - `/dropdown-funds`
  - `/dropdown-expense-types`
  - `/dropdown-vendors`
  - `/dropdown-product-ins/{productId}` (or backend equivalent)
- Serials for a product-in batch: `/product-in-serials/{productInId}`

If any dropdown endpoint name differs on your backend, adjust only the `api:` strings; keep the UI + typing intact.
