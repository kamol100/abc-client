# Plan 03 — Product details + reports + remaining integrations

## 🧠 Architecture explanation

Plan-01 gave us the Product master list + CRUD. Plan-02 gave us Product In/Out creation with line editors.

This plan completes the feature by migrating:

- **Per-product movement details** pages:
  - `/products/in/[productId]` and `/products/out/[productId]` — show the paginated list of Product In/Out rows for a given product, with filters, totals summary, serial details, and delete action (permission-gated).
- **Reporting**:
  - `/products/reports` — a combined “In vs Out” report page with filters and summary metrics.
  - `/reports/product-in` and `/reports/product-out` — report pages under the main Reports menu for parity with old app.
- **Serial details UI**:
  - Replace Flowbite `Badge/Modal/Table` serial popover with shadcn `Badge/Dialog/Table`, reusable across details + reports tables.

Implementation principles (from `migrate-feature.md`):

- Use **DataTable + FormFilter** (already used by invoices) instead of old `FilterForm`.
- Use **strict Zod schemas** (`*.passthrough()` for API rows) and **no `any`**.
- No hardcoded user-facing strings; all labels/headings/buttons/errors via i18n keys.
- Permissions via `usePermissions().hasPermission(...)`.

## 🧾 All new TypeScript types (strict Zod)

### Movement rows (details pages)

Old reference (behavior/columns): `[C:/wamp64/www/isp-client/components/products/product-view.tsx](C:/wamp64/www/isp-client/components/products/product-view.tsx)`

Create `[components/products/product-movement-type.ts](C:/wamp64/www/shadcn-isp-client/components/products/product-movement-type.ts)` with:

- **Serial item schema** `ProductSerialItemSchema`:
  - `id`, `serial_number`, `purchase_date`, `out_date` (nullable)
- **Ref schemas**:
  - `ProductRefSchema` (id, name)
  - `StaffRefSchema` (id, name)
  - `VendorRefSchema` (id, name)
  - `PurchaseByRefSchema` / `ReceivedByRefSchema` (id, name)
- **Row schema** `ProductMovementRowSchema` with `.passthrough()`:
  - shared fields used by both in/out rows:
    - `id`, `status`, `quantity`, `quantity_remaining`, `unit_price`, `total_price`, `unit_sell_price`, `note`
    - `purchase_date` and/or `out_date` (nullable)
    - `product` (ProductRef)
    - `vendor` (VendorRef nullable)
    - `purchaseBy` / `receivedBy` (nullable)
    - `staff` (nullable)
    - `serial: ProductSerialItemSchema[]` (default empty)
- **List response shape**:
  - `ProductMovementReportsSchema` for the summary numbers (support both in/out keys):
    - `product_in_current_total_price`, `product_in_current_quantity`
    - `product_out_current_total_price`, `product_out_current_quantity`
  - `ProductMovementListApiResponse` typed like invoices use:
    - `ApiResponse<{ data: ProductMovementRow[]; pagination: Pagination; reports?: ProductMovementReports | null }>`

  - `mode === "in"` → `useApiQuery({ url:` products-in/${productId}`, queryKey: ["products-in", productId], params })`
  - `mode === "out"` → `useApiQuery({ url:` products-out/${productId}`, queryKey: ["products-out", productId], params })`

  - Use DataTable toolbar’s `FormFilter` pattern (like invoices):

    - for out mode: client dropdown (`/dropdown-clients`) if supported
- Summary header:
  - Use `reports` block to display current total price + current total quantity (old UI shows 2 metrics).
- Actions:
  - Delete button via `DeleteModal`:
    - For in mode: `api_url={`/products-in/${row.id}`}`, permission `products-in.delete`
    - For out mode: `api_url={`/products-out/${row.id}`}`, permission `products-out.delete`
  - Invalidate keys should include: `products` + the specific movement query key.

Columns (match old intent, but modernized):

- Product name + status
- Purchaser/Out details cell: date + by (purchaseBy/receivedBy)
- Vendor
- Quantity
- Remaining
- Unit price
